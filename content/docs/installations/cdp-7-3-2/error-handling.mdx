This is the guide's troubleshooting log: the errors that actually surfaced while building this deployment, paired with what resolved them. It is organised by cluster — Base first, then Data Services (ECS) — and within each, roughly in the order the failures tend to appear.

Two themes run through the whole section. On the Base cluster, most failures are **certificate, Kerberos or PostgreSQL** problems wearing a service's name. On the ECS cluster, most failures trace back to an **incomplete cleanup** or **DNS that was not in place before the install** — which is why [Cluster Cleanup](/installations/cdp-7-3-2/cleanup) is worth doing properly rather than quickly.

## Base cluster error handling

### Preliminary commands

Two commands the guide keeps to hand: pointing `python` at Python 2 for older tooling, and pulling the Cloudera Manager server certificate out of the TLS handshake so it can be trusted elsewhere.

```bash
alternatives --set python /usr/bin/python2

openssl s_client -connect cldr-mngr.cldrsetup.local:8443 < /dev/null | sed -ne '/-BEGIN CERTIFICATE-/,/-END CERTIFICATE-/p' > knoxssoAmbari.crt
```

### Reports Manager

If you are using PostgreSQL, turn off Readline support by using the `-n` option. (history-passwd) Start the Cloudera Management Service when the Reports Manager role is ready. See *Starting the Cloudera Management Service*.

Ensure that the Ranger Solr and Ranger HDFS plugins are enabled.

<Banner type="important">
Ensure you complete the following tasks before you start performing the steps to configure TLS 1.2 on the Reports Manager for communicating with the database.
</Banner>

On the Cloudera Manager UI, navigate to **Clusters > Cloudera Management Service**. Select the **Configuration** tab and search for `reportsmanager_db_safety_valve`. Based on your database type you must override the `headlamp.db.properties` file with JDBC URL properties. Enter the appropriate values in the following format to override the connection to use TLS 1.2.

```ini
PostgreSQL
com.cloudera.headlamp.orm.hibernate.connection.url=jdbc:postgresql://<DB-HOST>:<DB-PORT>/<DB_NAME>?useSSL=true&trustCertificateKeyStoreUrl=<PATH_TO_TRUSTSTORE_FILE>&trustCertificateKeyStoreType=<TRUSTSTORE_TYPE>&trustCertificateKeyStorePassword=<TRUSTSTORE_PASSWORD>
com.cloudera.headlamp.db.type=postgresql
com.cloudera.headlamp.db.host=<DB-HOST>:<DB-PORT>
com.cloudera.headlamp.db.name=<DB_NAME>
```

### Ozone client config issue while deploying Kerberos

A known issue. It appears that you might have a proxy set up for the Administration Console. Specify the proxy URL as the Frontend URL, or disable the **HTTP Referer Check** option.

### Ranger and Atlas not running

Due to a Kafka issue and a Solr issue — fix those first, below.

### Solr error

Initialize Solr and create the HDFS home dir from **Actions**, then start the service. That will fix the issue.

### Kafka error (dependency order)

Kafka and Solr depend on Ozone (and Solr depends on Kafka as well) — install Ozone first.

### Tez error

In Cloudera Manager: **Tez > Actions > Upload Tez file to HDFS**, then **Hive > Actions > Create Hive dir**.

### YARN Queue Manager error

The Queue Manager store directory is missing or owned by the wrong account:

```bash
[root@pvcbase-master ~]# sudo mkdir /var/lib/hadoop-yarn/
[root@pvcbase-master ~]# sudo chmod +077 /var/lib/hadoop-yarn/
[root@pvcbase-master ~]# sudo chown yarn:hadoop /var/lib/hadoop-yarn/
```

### Kafka error (stale cluster id)

**Solution.** Delete `/var/local/kafka/data/meta.properties`. This file pins the broker to a cluster id from the previous install; removing it lets the broker rejoin the new cluster.

### Enable the Thrift server for HBase and Hue

```text
set wal property codec-hbase
```

### HBase master bad health

The problem lies in the Cloudera Management Monitor service, not in HBase itself. Restart the Cloudera Management Monitor service, then restart HBase. After that everything should be fine.

### Ozone error: "Could not find or is not a file"

Make sure that `hdfs_service` is enabled in the Ozone configuration. With this enabled, the Cloudera Manager agent will put `core-site.xml` into the process directory and the error will be gone. Clean up the Ozone directories before redeployment.

### Resetting the Ranger admin password in PostgreSQL

Ranger stores the admin password as an MD5 hash in `x_portal_user`. Generate the hash for the password you want, then update the row directly. Verify first that you can reach PostgreSQL over TLS at all:

```text

Password for user postgres:
psql (17.11)
SSL connection (protocol: TLSv1.2, cipher: ECDHE-RSA-AES256-GCM-SHA384, bits: 256, compression: off)
Type "help" for help.

postgres=# \q

[root@pvcbasemaster data]# echo -n 'Sahu@123{admin}' | md5sum
c94251c29cd07ed2daf0b6edcf843362  -
```

Then update the row and read it back:


### Atlas and Knox UI login failures

**Solution.** You need to update permissions in order to access the Atlas and Knox UIs with the default PAM authentication enabled:

```bash
chmod 444 /etc/shadow
```

World-readable `/etc/shadow` exposes every local password hash on the host. This is the guide's lab workaround; on any real deployment, configure PAM or LDAP authentication properly instead.

### Dashboard token: the documented command returns nothing

The `sa/admin-user` service account no longer carries a `secrets[0]` reference on this Kubernetes version:

```bash
kubectl -n kubernetes-dashboard get sa/admin-user -o 'jsonpath={.secrets[0].name}'
```

**Solution.** Read the secret by name instead:

```bash
kubectl -n kubernetes-dashboard get secret/admin-user-secret  -o 'jsonpath={.metadata.name}'
(working)
```

### Dashboard token: `base64decode` template error

Without naming the secret, the template runs against a list and `base64decode` gets a non-string:

```text
+ /var/lib/rancher/rke2/bin/kubectl --kubeconfig /etc/rancher/rke2/rke2.yaml -n
kubernetes-dashboard get secret -o 'go-template={{.data.token | base64decode}}'
error: error executing template "{{.data.token | base64decode}}": template: output:1:16: executing
"output" at <base64decode>: invalid value; expected string
```

**Solution.**

```bash
kubectl get secret -n kubernetes-dashboard admin-user-secret -o 'go-template={{.data.token | base64decode}}' (working command)
```

### CDW Data Catalog creation fails on a certificate

```text
Initializing the schema to: 3.1.3000.2025.0.19.1-49
Metastore connection URL:
jdbc:postgresql://postgres-service-default-warehouse:5432/hive?createDatabaseIfNotExist=true&sslmode=verify-ca&sslrootcert=/mnt/certs/hms-root.crt
Metastore connection Driver: org.postgresql.Driver

Underlying cause: org.postgresql.util.PSQLException : SSL error: PKIX path building failed:

│ warehouse-cldrsetup                  metastore-0                    ●   0/1                     Init:0/4                    0   0      0
0     n/a       0                      0 n/a    pvcecs-worker10.cldrsetup.local                     20s          │
```

**Solution.** The CDW Data Catalog creation failed due to a certificate validation error between the Hive metastore and the PostgreSQL backend. This was caused by an invalid or missing truststore certificate (`hms-root.crt`). To fix this, extract the correct certificate, validate it, and reapply the secret before restarting the metastore pod.

```bash
# Backup current file if needed
kubectl get secret pg-hms-db-certs -n warehouse-cldrsetup -o yaml > pg-hms-db-certs-backup.yaml

# Recreate the secret with the expected key
kubectl delete secret pg-hms-db-certs -n warehouse-cldrsetup

kubectl create secret generic pg-hms-db-certs \
  --from-file=TRUSTSTORE_PEM=root.crt \
  -n warehouse-cldrsetup
```
