Before getting into the actual installation of Cloudera on premises Base and Data Services clusters, we need to prepare our machines and perform some steps to meet the prerequisites.

Choose one of the nodes of the cluster or a separate node as the Ansible Admin/Controller Node for management. In this document, we configured the `ipaserver` for this purpose.

---

> **Note:** By default, running `dnf update` on RHEL may upgrade the system to the latest minor release (for example, 9.6). If you want to stay on RHEL 9.5 and receive only security and bug-fix updates, lock the system to your current minor version.

Solution 1 - Lock to your current minor version (RHEL 9.5). Run the following:

```bash
sudo subscription-manager release --set=9.5
```

Then verify:

```bash
sudo subscription-manager release --show
```

Expected output:

```text
Release: 9.5
```

This locks your system repositories to RHEL 9.5 content - updates will stay within 9.5 and not move to 9.6.

## 3. Set up password-less login

To manage all the nodes in a cluster from the admin/controller node, password-less login needs to be set up. It assists in automating common tasks with Ansible and shell scripts without having to use passwords.

Enable password-less login across all the nodes when Red Hat Linux is installed across all the nodes in the cluster.

**Step 1.** Log in to the `ipaserver` node.

```bash
[root@ipaserver ~]# ssh 172.31.24.240
```

**Step 2.** Run the `ssh-keygen` command to create both the public and private SSH key pair on the Ansible controller node.

```bash
[root@ipaserver ~]# ssh-keygen -N '' -f ~/.ssh/id_rsa
[root@ipaserver ~]# ls -l /root/.ssh
[root@ipaserver ~]# chmod 600 /root/.ssh/id_rsa
```

![Terminal running ssh-keygen -t rsa -f /root/.ssh/id_rsa_new and prompting for a passphrase](/images/cdp-7-3-2/post-os-work/ssh-keygen-prompt.png)

**Step 3.** Run the following commands from the Ansible controller / `ipaserver` node to copy the public key `id_rsa.pub` to all the nodes of the cluster. `ssh-copy-id` appends the keys to the remote hosts' `.ssh/authorized_keys`.

> Not applicable on AWS EC2 - this needs to be done manually, as password-based authentication for non-root users is disabled there.

```bash
[root@ipaserver ~]# for i in {1}; do echo "copying ipaserver.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@ipaserver.cldrsetup.local; done;
[root@ipaserver ~]# for i in {1}; do echo "copying cldr-mngr.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@cldr-mngr.cldrsetup.local; done;
[root@ipaserver ~]# for i in {1}; do echo "copying pvcbase-master.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@pvcbase-master.cldrsetup.local; done;
[root@ipaserver ~]# for i in {1..3}; do echo "copying pvcbase-worker$i.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@pvcbase-worker$i.cldrsetup.local; done;
[root@ipaserver ~]# for i in {1}; do echo "copying pvcecs-master.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@pvcecs-master.cldrsetup.local; done;
[root@ipaserver ~]# for i in {1..10}; do echo "copying pvcecs-worker$i.cldrsetup.local"; ssh-copy-id -i
~/.ssh/id_rsa.pub root@pvcecs-worker$i.cldrsetup.local; done;

##### Alternate way is to add pub key to authorized_keys file manually on ipaserver node and copy the entire
.ssh directory to all other NODES; otherwise login into each hosts and manually update authorized_keys:

[root@ipaserver ~]# cat /root/.ssh/id_rsa.pub >> /root/.ssh/authorized_keys

##### copy the entire .ssh directory to all NODES
[root@ipaserver ~]# scp -r /root/.ssh root@cldr-mngr.cldrsetup.local:/root/.
##### (provide root user password when prompted)

##### Download the id_rsa and id_rsa.pub to your local machine by either using scp or sftp (as it will be
required later)
```

**Step 4.** Enter `yes` for *Are you sure you want to continue connecting (yes/no)?*

**Step 5.** Enter the password of the remote host.

> **Note:** Make sure that the `/etc/resolv.conf` file on the ECS hosts contains a maximum of 2 active search domains. See [Known issues in ECS environments](https://docs.cloudera.com/data-warehouse/1.5.5/release-notes/topics/dw-private-cloud-known-issues-ecs-environments.html).

## 10. Set ulimit

## 15. Disable Transparent Huge Pages

Disabling Transparent Huge Pages (THP) reduces elevated CPU usage caused by THP.

**Step 1.** You must run the following commands for every reboot.


```bash

[root@jump ~]# echo "echo never > /sys/kernel/mm/transparent_hugepage/defrag" >> /root/thp_disable

[root@jump ~]# echo "sysctl -w net.ipv6.conf.all.disable_ipv6=1" >> /root/thp_disable
[root@jump ~]# echo "sysctl -w net.ipv6.conf.default.disable_ipv6=1" >> /root/thp_disable
[root@jump ~]# echo "sysctl -w net.ipv6.conf.lo.disable_ipv6=0" >> /root/thp_disable
```

**Step 3.** Copy the file to each node to copy the commands to `/etc/rc.d/rc.local`, so they are executed automatically for every reboot.

## 17. Turn off TCP checksum offload on the ECS nodes

This applies to the network interface of all ECS nodes, both master and worker, and is **only applicable for VMware machines**.

The default CNI (Canal) that comes with ECS (RKE2 from SUSE) does not support VMs. Customers have reported escalations with the default CNI when they use VMware. Customers may experience network connectivity issues or degraded performance when deploying ECS clusters on VMware environments with the default CNI, so they use a workaround by turning off TCP checksum offload on the interface.

> **Note:** Disabling TCP checksum offload may have implications on network performance or security. Evaluate the impact of this workaround in the specific environment and consider reverting the changes once a permanent solution or alternative workaround is available. Additionally, consult with the VMware documentation or support resources for guidance on network configuration and optimization in VMware environments.

**Step 1.** Identify the network interface.

```bash

the `ifconfig` or `ip addr` command:
[root@jump ~]# ansible ecsmasternodes,ecsnodes -m shell -a "ifconfig | grep flags | grep -v lo && ip addr |
grep mtu | grep -v lo"

[root@jump ~]$ ifconfig | grep flags | grep -v lo

2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 9001 qdisc mq state UP group default qlen 1000

```

**Step 2.** Disable TCP checksum offload. Use the appropriate commands or configuration settings to disable TCP checksum offload on the identified network interface. For example, using `ethtool`:


## 19. Install httpd on the Cloudera Manager node

Install httpd on the Cloudera Manager node - `cldr-mngr` - to host a local parcel repository. Setting up the RHEL repository on the Cloudera Manager node requires httpd.

**Step 1.** Install httpd on the Cloudera Manager node (`cldr-mngr`) to host the repositories.

> **Note:** The Red Hat repository is hosted using HTTP on the admin node; this machine is accessible by all the hosts in the cluster.

```bash
[root@cldr-mngr ~]# dnf install -y httpd mod_ssl createrepo_c
```

**Step 2.** Generate the CA certificate.

```bash
[root@cldr-mngr ~]# openssl req -newkey rsa:2048 -nodes -keyout /etc/pki/tls/private/httpd.key -x509 -days
3650 -out /etc/pki/tls/certs/httpd.crt \
-subj "/C=US/ST=North Carolina/L=Raleigh/O=Red Hat Inc/OU=CLDR/CN=cldr-mngr.cldrsetup.local"

[root@cldr-mngr ~]# ls -l /etc/pki/tls/private/ /etc/pki/tls/certs/
/etc/pki/tls/certs/:
total 8

```

**Step 3.** Create the certificate directory to serve content from.

```bash
[root@cldr-mngr ~]# mkdir -p /var/www/https/
[root@cldr-mngr ~]# echo secure content > /var/www/https/index.html
[root@cldr-mngr ~]# cat /var/www/https/index.html
secure content
```

**Step 4.** Edit the `httpd.conf` file; add `ServerName` and make the necessary changes to the server configuration file.

```bash
[root@cldr-mngr ~]# vi /etc/httpd/conf/httpd.conf
ServerName cldr-mngr.cldrsetup.local:80
```

**Step 5.** Start the httpd service.

```bash
[root@cldr-mngr ~]# systemctl start httpd
[root@cldr-mngr ~]# systemctl enable httpd
[root@cldr-mngr ~]# systemctl is-enabled httpd
```
