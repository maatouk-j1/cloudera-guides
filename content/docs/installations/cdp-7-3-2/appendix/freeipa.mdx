FreeIPA underpins DNS, Kerberos and LDAP for the whole deployment — the base cluster's Kerberos realm, the reverse DNS records the ECS ingress depends on, and the LDAP directory that Cloudera Manager, Ranger, Hue, Atlas and the Data Services console all authenticate against. This page collects the commands used against it.

The guide's own reference for this section is [https://www.devopszones.com/2020/03/how-to-add-freeipa-user-in-cli-and-web.html](https://www.devopszones.com/2020/03/how-to-add-freeipa-user-in-cli-and-web.html).

## Add users on FreeIPA

- Log in to the IPA server and run kinit with admin and enter the password: **`kinit admin`**
- Run the below command to create a user. Enter the password in the password prompt.

```bash
ipa user-add --password <User Name>
```

## Create a user and generate its keytab

The same four steps, written out for a user named `cdptest`:

```bash
# 1. Add the user 'cdptest'
ipa user-add cdptest \
     --first=cdp \
     --last=test \
     --password
# (You'll be prompted to set and confirm a password for cdptest.)

# 2. Authenticate as an IPA admin
kinit admin

# 3. Generate a keytab for cdptest@CLDRSETUP.LOCAL
ipa-getkeytab \
  --principal=cdptest@CLDRSETUP.LOCAL \
  --keytab=/tmp/cdptest.keytab

# 4. Verify the keytab contents
klist -k /tmp/cdptest.keytab
```

## Service principal variant

If instead you need a **service principal** named `cdptest` (for example, to run as an SPN on a host), you would:

```bash
# Create a service principal
ipa service-add cdptest/your-host.cldrsetup.local

# Pull its keytab
ipa-getkeytab \
  --principal=cdptest/your-host.cldrsetup.local@CLDRSETUP.LOCAL \
  --keytab=/tmp/cdptest.keytab

# Verify
klist -k /tmp/cdptest.keytab
```

## List and slice keytabs

Commands to list the principals in a keytab and extract individual keytab entries:

```bash
# 1. List all principals in the keytab
klist -k /tmp/cdptest.keytab

# Example output:
# Keytab name: FILE:/tmp/cdptest.keytab
# KVNO Principal
# ---- ------------------------------------------------------
#    1 cdptest@CLDRSETUP.LOCAL

# 2. Extract a specific principal's keys into a new keytab
#    (useful if you want to split out one principal from a multi-principal keytab)
ktutil <<EOF
  rkt /tmp/cdptest.keytab
  l
  wkt /tmp/cdptest-single.keytab Principal:cdptest@CLDRSETUP.LOCAL
  quit
EOF

# 3. Verify the newly created single-principal keytab
klist -k /tmp/cdptest-single.keytab
```

`klist -k` shows you all principals stored in a keytab file. `ktutil` can read (`rkt`) an existing keytab, list (`l`) its entries, and write (`wkt`) only the entries matching a given principal into a new keytab. This way you can both inspect and slice your keytab as needed.

## Free-IPA Command Reference

```bash
kinit admin
ipa dnsrecord-add 16.172.in-addr.arpa. 231.31 --ptr-rec console-cdp.apps.pvcecsmaster.cldrsetup.local.
ipa dnsrecord-del 16.172.in-addr.arpa. 231.31 --ptr-rec console-cdp.apps.pvcecsmaster.cldrsetup.local
ipa dnsrecord-find 16.172.in-addr.arpa.
ipa dnsrecord-add cldrsetup.local *.apps
ipa dnszone-list
ipa user-del cmadmin
ipa user-show
ipa status | start | stop
ipactl status | stop | start | restart
```

The last two lines are not pipelines — they are the guide's shorthand for the subcommand each takes. Run `ipactl status`, `ipactl restart` and so on individually.

## First login and password expiry

A user created with `--password` gets a pre-expired password, so the first `kinit` forces a change:

```text
[kuldeep@pvcbasemaster ~]$ kinit kdsahu
Password for kdsahu@CLDRSETUP.LOCAL:
Password expired.  You must change it now.
Enter new password:
Enter it again:
[kuldeep@pvcbasemaster ~]$
```

## Source quirks in this section

This section is visibly less finished than the rest of the guide — parts of it are pasted assistant output, complete with stray `bash` fence markers and emoji step numbers, which have been dropped here in favour of plain numbering. Four things are worth knowing before following it literally:

| What the source shows | The discrepancy |
|:--|:--|
| *"Log in to the IPA server"*, then a transcript prompted `[root@pvcbasemaster ~]#` | The worked example runs from the **base cluster master**, not from `ipaserver` — the host the instruction names. The `ldapsearch` example further down does run on `[root@ipaserver ~]#`. |
| `ipa dnsrecord-find` listing **five** PTR records | The summary beneath it reads `Number of entries returned 12`. The listing is abridged; the zone holds twelve records. |
| `krbExtraData:: AAK1j/Blcm9vdC9hZG1pbkBDRFBQVkNEUy5DT00A` | Base64-decodes to `root/admin@CDPPVCDS.COM` — a principal from the older `cdppvcds.com` lab build, in a directory whose every other attribute reads `cldrsetup.local`. Same recycled-lab quirk seen in the screenshots throughout the guide. |
| `memberOf: cn=Replication Administrators,…,dc=cldrsetup,dc=c om` | Unfolds to `dc=cldrsetup,dc=com`, where all twelve sibling entries end `dc=cldrsetup,dc=local`. |

The `ldapsearch` bind password `cloudera123` is the guide's lab value and is transcribed verbatim; it is not a credential to reuse.
