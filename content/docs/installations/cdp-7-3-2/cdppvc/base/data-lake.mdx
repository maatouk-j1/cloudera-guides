This page continues the procedure begun on [Install CDP PvC Base using the CM Web UI](/installations/cdp-7-3-2/cdppvc/base/installation). The runtime is installed and the cluster inspected; clicking **Continue** at the end of Step 17 hands straight over to the **Add Cluster - Configuration** wizard, which runs in nine stages - Select Services, Assign Roles, Setup Database, Enter Required Parameters, Review Changes, Configure Kerberos, two Command Details stages and Summary. The step numbering picks up at **Step 18**.

## 1. Select the services

**Step 18.** After the Cloudera on premises Base Cluster (runtime) setup is complete, if successful, it will automatically move to add services *(Add Cluster -Configuration)* wizard. It will ask to **Select Services** i.e. (a) Data Engineering, (b) Data Warehouse/ Data Mart, © Operational Database specific or (d) Custom control plane/base cluster services i.e. HDFS, YARN, Hive, Tez, HiveOnTez, Ozone, Zookeeper, Kafka, SOLR, Ranger, Atlas, HBase, Phoenix, Impala, HUE, Spark2, Spark3, YARN Queue Manager etc. Choose from a combination or services defined or select custom services. Services required based on selection will be automatically added.

**Step 19.** Select **Custom Services** option to install.

![Select Services stage of the Add Cluster - Configuration wizard, offering Data Engineering, Data Mart, Operational Database and Custom Services, with Custom Services selected and the nine wizard stages listed down the left](/images/cdp-7-3-2/cdppvc/base/data-lake/select-services.png)

> **Note:** It is important to select host(s) to deploy services based on the role intended for it. For detailed information, go to: [Runtime Cluster Hosts and Role Assignments](https://docs.cloudera.com/cdp-private-cloud-base/7.1.9/installation/topics/cdpdc-runtime-cluster-hosts-role-assignments.html)

**Step 20.** Under the **Custom Services**, select the below custom Cloudera DataLake/Control Plane management services to be installed on the cluster. The Selection would look similar to below, Select services and Click **Continue**.

![Custom Services list, first part, with Atlas, HBase, HDFS, Hive, Hive on Tez, Hue and Iceberg Replication ticked and Cruise Control left unticked](/images/cdp-7-3-2/cdppvc/base/data-lake/custom-services-1.png)

![Custom Services list continuing through Impala, Kafka, Knox ticked, with Key-Value Store Indexer, Kudu, Livy and Livy for Spark 3 left unticked](/images/cdp-7-3-2/cdppvc/base/data-lake/custom-services-2.png)

![Custom Services list continuing with Ozone, Phoenix, Ranger, Solr, Spark and Spark 3 ticked, and Query Processor, SQOOP_CLIENT, Schema Registry, Streams Messaging Manager, Streams Replication Manager and Stub DFS left unticked](/images/cdp-7-3-2/cdppvc/base/data-lake/custom-services-3.png)

## 3. Set up the databases

**Step 23.** Click **Continue**. When you're doing the set-up for the first time on this CM Sever. **Cloudera Management Services** will be installed only once along with other control plane services.

**Step 24.** On the **Setup Databases** page, Select **database type** as **Use Custom Database**. Provide database hostname, username, and password (created in earlier steps) for different services and click on **Test Connection**. After a successful connection test, click **Continue** to install, configure and start services sequentially.

- Reports Manager (For Cloudera Management Service)
- Oozie Server (If selected to install, as optional)
- Ranger
- Hive
- YARN Queue Manager
- Hue

The databases and roles themselves were created back on the [Setup Database for Cloudera Manager](/installations/cdp-7-3-2/cdppvc/cm/database) page.

![Setup Database stage with Reports Manager, Ranger, Hive and Hue each set to PostgreSQL against the Cloudera Manager host, every block reporting Successful](/images/cdp-7-3-2/cdppvc/base/data-lake/setup-database.png)

## 4. Enter the required parameters

**Step 25.** Next, the **Enter Required Parameters** page appears.

**Step 26.** Wait at the parameter screen, enter the required parameters. For all the remaining parameters, set a common password so that it becomes easier while using those services. This password must have 1 lowercase, 1 uppercase, and 1 numeric value. Failing to adhere to this, the final step in the cluster setup would fail. Please see the required inputs below:

- *Knox Master Secret:* **`Cloudera@123`**
- *Knox IDBroker Master Secret:* **`Cloudera@123`**
- *Enter a suitable name for Ozone Service ID. i.e.:* **`ozone11`**
- Ranger Admin: **`Cloudera@123`**
- Ranger Usersync: **`Cloudera@123`**
- Ranger Tagsync: **`Cloudera@123`**
- Ranger KMS Keyadmin: **`Cloudera@123`**

![Enter Required Parameters stage listing Knox Master Secret, Ozone Service ID set to ozone11, and the Ranger Admin, Usersync, Tagsync and KMS Keyadmin initial passwords, with a 1 - 6 of 6 row count](/images/cdp-7-3-2/cdppvc/base/data-lake/enter-required-parameters.png)

> **Note on the capture:** the guide lists **seven** inputs but the screenshot shows **six** (`1 - 6 of 6`) - Knox IDBroker Master Secret is absent from the capture. It appears only when the Knox IDBroker role is part of the selection.

**Step 27.** Click **Continue**.

## 5. Review the changes

**Step 28.** The **Review Changes** page appears. Set a **password** for **Atlas**. You can set it to the same value set for Ranger above.

- Atlas Admin password: **`Cloudera@123`**

![Admin Password row on the Review Changes page, showing the atlas.admin.password property for Cluster 1 - Atlas Server Default Group with a masked value entered](/images/cdp-7-3-2/cdppvc/base/data-lake/atlas-admin-password.png)

**Step 29.** Scroll down and verify/update the HDFS disks configuration according to the below.

- DataNode Data Directory: **`/hdfs/dfs/dn`**
- NameNode Data Directories: **`/hdfs/dfs/nn`**
- HDFS Checkpoint Directories: **`/hdfs/dfs/snn`**

![Review Changes page filtered to HDFS, showing HDFS Block Size 128 MiB, DataNode Failed Volumes Tolerated 0, and the three HDFS directory properties set to /hdfs/dfs/dn, /hdfs/dfs/nn and /hdfs/dfs/snn, with the TLS keystore location resolving to the CM_AUTO_TLS macro](/images/cdp-7-3-2/cdppvc/base/data-lake/review-changes-hdfs-directories.png)

**Step 30.** Review the changes for all the services on the **Review Changes** page and verify/edit the configuration parameters as per your requirements. Click **Continue**.

**Step 31.** A few sets of commands are running in the background. Wait for them to get executed successfully. Once done, Click **Continue**.

## 6. Configure Kerberos

**Step 32.** Configure Kerberos and Keep Review and customize the configuration changes based on your requirements. Check the box for **Enable Kerberos for this cluster**. Required libraries i.e. `krb5-workstation`, `krb5-libs` and `freeipa-client` are already installed on all servers in prior steps.

![Configure Kerberos stage with Enable Kerberos for this cluster ticked, the per-OS client library install commands listed, and Configure DataNode Ports showing DataNode Transceiver Port 1004 and DataNode HTTP Web UI Port 1006](/images/cdp-7-3-2/cdppvc/base/data-lake/configure-kerberos.png)

The client libraries were installed during [Post OS Installation - Preliminary Work](/installations/cdp-7-3-2/post-os-work), and the KDC itself was wired into Cloudera Manager on [Enable Kerberos](/installations/cdp-7-3-2/cdppvc/cm/kerberos).

**Step 33.** Click **Continue** after Cloudera Manager successfully runs the **Enable Kerberos** command.

![Command Details for the Enable Kerberos Command, Finished against PvCBaseCluster1, reporting Successfully enabled Kerberos and 7 of 7 steps completed - stop cluster, stop Cloudera Management Services, deploy krb5.conf, configure all services to use Kerberos, wait for credentials, deploy client configuration and start Cloudera Management Services](/images/cdp-7-3-2/cdppvc/base/data-lake/enable-kerberos-command.png)

## Source values that do not match this build

Every capture in this procedure comes from the same older lab session as the ones on the [installation page](/installations/cdp-7-3-2/cdppvc/base/installation), so the values inside them disagree with the prose beside them. The prose and the [version table](/installations/cdp-7-3-2/summary) win.

| Where | What the screenshot shows | What this build actually is |
|:--|:--|:--|
| Wizard chrome, every capture | Cloudera Manager `7.11.3` | Cloudera Manager **7.13.2.0** |
| Every host field in Assign Roles and Setup Database | `pvcbase-*.cdppvcds.com`, `cldr-mngr.cdppvcds.com` | `pvcbase-master`, `pvcbase-worker[1-3]` and `cldr-mngr` on `cldrsetup.local`, as the role table above states |
| Host counts in Assign Roles | `× 5 New` workers, `pvcbase-worker[1-5]` | three workers, `pvcbase-worker[1-3]` |
| CM Home after Step 39 | `Cloudera Runtime 7.1.9 (Parcels)`, `6 Hosts` | Runtime **7.3.2.0** on the four-node Base cluster |
| Ozone S3 Gateway in Assign Roles | `pvcbase-worker4` | `pvcbase-master`, as the role table states |
| Browser address bar | an AWS ELB hostname, `...us-west-2.elb.amazonaws.com:7183` | `https://cldr-mngr.cldrsetup.local:7183/cmf/` |

The role table itself is not internally consistent either, and is transcribed as printed rather than reconciled: the Kafka row names `base-master` and `pvcbaseworker[1-5]` where every other row uses `pvcbase-master` and `pvcbase-worker[1-3]`; the ZooKeeper row is headed *"must be &gt;3"* but lists three hosts while the screenshot assigns a single `Server × 1 New`; and the `YARN Queue Manager (Optional)` row has an empty Host cell.

The **Runtime Cluster Hosts and Role Assignments** link in Step 19 carries no version in its displayed text, and its underlying annotation targets the **7.1.9** documentation - it is linked as the source has it. The links in the Step 35 troubleshooting block match their displayed text and are wired up as such.

## What comes next

The Base cluster is up and healthy, but a set of service-level adjustments still has to be made before it is usable - Web UI access, port whitelisting and host-file entries. See [Additional Requirements and Details](/installations/cdp-7-3-2/cdppvc/base/additional-requirements).
